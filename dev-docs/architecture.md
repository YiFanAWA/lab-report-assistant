# 实验报告助手｜架构设计

> 依据：[project-charter.md](project-charter.md)  
> 阶段约束：后续每个新切片必须先确认对应 SPEC，再进入实现。  

## 架构决策

本项目采用单一推荐架构：**本地单用户 Web MVP + 模块化单体后端 + 独立 Worker + 应用托管的 Python 受控执行环境 + 统一大模型网关 + 证据化工作流核心**。

核心判断：

- 第一版先做一个可运行的 Web 工作台，不做 App、小程序、多人协作和自由拖拽工作流。
- 第一版部署形态锁定为本地单用户版本，不做在线多用户账号体系。
- 第一版不做注册登录。
- 第一版首个标准演示课题锁定为“胃病数据分析”。
- 业务语义由后端工作流核心归属层负责，前端只展示状态、收集输入、触发命令和展示结果。
- URL 采集、文档解析、LLM 调用、Python 执行、图表生成和 Word/PPT 生成由独立 Worker 承担，不长期阻塞普通 HTTP 请求。
- 大模型只能通过统一网关参与需求拆解、证据提取、方案生成、代码辅助和交付物文本生成；它不拥有最终业务真相。
- Python 执行必须由应用托管的受控执行环境承担，用户不需要手动配置 Python 环境或安装分析依赖。
- V1 Python 执行采用独立进程、独立工作目录、固定依赖白名单、执行超时、文件访问范围和输出大小限制；在线多用户版本再升级为容器级或等价隔离。
- Word 与 PPT 必须从同一份已确认实验大纲和真实执行结果生成，不能各自直接从模型临时上下文生成。
- V1 大模型供应商暂定 DeepSeek，默认模型为 `deepseek-v4-pro`，通过 `LLMGateway` 和配置接入，后续可经决策记录替换。

## 技术栈决策

技术栈已由项目负责人确认并锁定，详见 [tech-stack.md](tech-stack.md)。该技术栈已用于 SPEC 0001，后续切片继续遵守同一主线；如新增依赖或改变运行方式，必须更新依赖复核或新增决策记录。

V1 主线：

- 前端：TypeScript + React + Vite，适合多工作区 Web MVP、表格、代码编辑、状态面板和文件下载。
- 前端路由与接口状态：React Router；TanStack Query。
- 后端：Python + FastAPI，便于统一处理文档解析、数据分析、Python 执行编排、LLM 网关和文件生成。
- 数据访问：Pydantic、SQLAlchemy、Alembic。
- 数据库：V1 使用 SQLite 作为本地单用户存储；若进入在线多用户部署，再通过决策记录切换到 PostgreSQL。
- 文件存储：第一版使用项目工作目录下的本地文件存储抽象，保存原始资料、数据版本、日志、图表、Word 和 PPT。
- 后台任务：数据库任务表 + 独立 Worker 进程，前端通过轮询获取任务进度，后续可扩展 SSE。
- Python 执行：应用托管的受控执行环境，固定依赖、限定工作目录、限制运行时间、记录 stdout/stderr 和产物清单；V1 不强制依赖 Docker。
- LLM：V1 暂定 DeepSeek，通过 `LLMGateway` 抽象供应商，不在业务层直接绑定某个模型名。

## 框架最佳实践合同

进入实现阶段后，必须遵守所选框架的目录、生命周期、类型、测试和构建实践。不得在已选框架旁边私造一套并行架构。

约束：

- 前端不私造业务状态机；只消费后端返回的工作流状态、任务状态、错误原因和产物索引。
- 后端不把 LLM prompt 当作业务合同；LLM 输出必须落入结构化 schema，再由规则校验。
- Python 执行器不直接读写任意项目文件；只能访问被授权的数据版本目录和输出目录。
- 文件生成器不从散落文本拼装最终结论；必须消费已确认大纲、证据卡片、执行记录和图表索引。
- Worker 不拥有业务语义；它只领取任务、调用核心服务、写回结构化状态、日志和产物索引。

## 唯一归属层

### 1. 项目工作区核心

唯一归属：实验项目、课题、版本、阶段状态、变更记录、项目文件归属。

它创建并维护 `Project`、`Topic`、`WorkflowState`、`ChangeRecord`。

### 2. 实验要求核心

唯一归属：实验要求来源、结构化任务单、未明确项、复刻层级 L0-L3、老师要求与模型建议的区分。

它创建并维护 `RequirementSource`、`RequirementPlan`、`ReplicationLevel`。

### 3. 来源与证据核心

唯一归属：公开 URL、本地资料、解析文本、证据卡片、来源位置、采集状态。

它创建并维护 `Source`、`ParsedDocument`、`EvidenceCard`。

### 4. 数据集与分析核心

唯一归属：数据集、数据版本、字段概览、质量检查、清洗方案、分析方案、图表方案。

它创建并维护 `Dataset`、`DatasetVersion`、`DataProfile`、`AnalysisPlan`。

### 5. 执行核心

唯一归属：Python 代码任务、执行请求、执行日志、表格输出、图表输出、失败状态。

它创建并维护 `CodeTask`、`ExecutionRun`、`ExecutionArtifact`。

### 5.1 后台任务核心

唯一归属：后台任务记录、任务状态、重试策略、取消状态、任务日志和 Worker 领取边界。

它创建并维护 `BackgroundJob`、`JobStatus`、`JobLog`。

### 6. 大纲与交付物核心

唯一归属：统一实验大纲、Word/PPT 生成请求、交付物版本、交付物与证据/执行记录之间的追溯关系。

它创建并维护 `Outline`、`Deliverable`、`DeliverableVersion`。

### 7. 大模型网关

唯一归属：模型调用、结构化输出、token/成本记录、供应商替换边界。

它不拥有项目业务状态，只返回可校验候选结果。

### 8. 界面适配层

唯一归属：页面展示、表单输入、用户操作、下载入口、用户确认动作。

它不拥有核心业务语义，也不私自判断任务完成、结果可信或医学结论。

## 核心合同

第一版需要优先定义以下合同，进入编码前应在实现计划中变成 schema 或类型：

- `ProjectStatus`：项目阶段、当前工作区、阻塞原因、可执行命令。
- `RequirementPlan`：必须任务、推荐任务、可选任务、超范围任务、未知项、验收条件。
- `ReplicationLevel`：L0、L1、L2、L3 超范围及说明。
- `SourceRecord`：URL/文件、类型、采集状态、标题、时间、错误原因、原始文件位置。
- `EvidenceCard`：摘要、证据类型、来源位置、确认状态。
- `DatasetVersion`：来源、版本号、字段、质量概览、文件位置。
- `AnalysisPlan`：清洗步骤、统计方法、图表计划、用户确认状态。
- `ExecutionRun`：代码版本、数据版本、状态、日志、表格、图表、错误。
- `BackgroundJob`：任务类型、任务状态、归属项目、输入引用、输出引用、错误原因。
- `Outline`：章节、关联证据、关联执行结果、确认状态。
- `Deliverable`：类型、版本、生成状态、下载位置、追溯索引。

## 数据流

```text
项目工作区核心
  -> 实验要求核心
  -> 来源与证据核心
  -> 数据集与分析核心
  -> 执行核心
  -> 大纲与交付物核心
```

大模型网关只在每个核心归属层请求时提供结构化候选结果。界面适配层只显示状态并提交用户确认。独立 Worker 只执行任务并写回核心合同，不拥有业务结论。

## 工作流状态

项目阶段状态、长任务状态和过期传播由后端核心拥有。前端不得私造状态机。

项目阶段至少覆盖：

```text
DRAFT
REQUIREMENT_PARSED
REQUIREMENT_CONFIRMED
SOURCES_COLLECTED
EVIDENCE_CONFIRMED
DATASET_READY
ANALYSIS_PLANNED
ANALYSIS_CONFIRMED
EXECUTING
EXECUTION_FAILED
RESULT_CONFIRMED
OUTLINE_CONFIRMED
GENERATING
COMPLETED
```

长任务状态至少覆盖：

```text
PENDING
RUNNING
SUCCEEDED
FAILED
CANCELLED
STALE
```

`STALE` 表示上游内容变化导致下游产物过期。用户修改实验要求、数据版本、分析方案或执行结果后，受影响的大纲、图表、Word 和 PPT 不得继续被视为有效产物。

## 追踪链

实验性结论必须能追踪到：

```text
报告结论
  -> 图表或表格
  -> ExecutionRun
  -> Python 代码版本
  -> 数据版本
  -> 原始数据集
```

资料性描述必须能追踪到：

```text
理论描述
  -> EvidenceCard
  -> 原文位置
  -> 来源文件或 URL
```

这条追踪链是第一版核心价值，不得被简化为只在最终文档里写引用。

## 禁止路径

- 禁止让界面判断实验结果是否真实。
- 禁止让提示词文本成为任务状态或验收结论的唯一来源。
- 禁止直接从原始大模型对话历史生成 Word/PPT。
- 禁止把 L1/L2 参考工作描述为完整论文复现。
- 禁止绕过登录、付费、验证码或访问限制获取资源。
- 禁止在 V1 引入注册登录、多人账号管理或云端协作。
- 禁止静默覆盖旧实验要求、方案、代码执行记录、图表或交付物。
- 禁止把医学教学数据集分析改写成诊断或治疗建议。
- 禁止让用户手动安装分析依赖作为 V1 正常路径。
- 禁止让普通 HTTP 请求长期承担采集、解析、执行或交付物生成任务。
- 禁止在业务模块中写死 DeepSeek 模型名或真实 API 密钥。
- 禁止在未确认对应 SPEC 的情况下初始化新框架、安装新依赖或编写新切片业务代码。

## 风险边界

- 网页采集风险：第一版只处理公开可访问 URL 和本地文件，失败必须结构化返回。
- 大模型风险：所有模型输出必须区分事实、建议、未知和超范围。
- 执行安全风险：V1 本地单用户采用受控独立进程、限时、限目录、固定依赖、限制输出和完整日志；在线多用户版本必须重新评估容器级隔离。
- 学术诚信风险：产品定位为辅助理解和生成可审阅草稿，不承诺未经检查直接提交。
- 医学主题风险：只做教学数据分析，不做个体诊断和治疗建议。
