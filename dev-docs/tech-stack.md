# 实验报告助手｜技术栈选择

> 状态：已确认锁定  
> 确认日期：2026-06-16  
> 依据：[project-charter.md](project-charter.md)、[architecture.md](architecture.md)  
> 阶段约束：本文档只锁定技术路线，不批准创建业务代码、安装依赖或初始化技术框架。

## 1. 产品形态

第一版产品形态锁定为：**本地单用户 Web MVP 工作台**。

选择理由：

- 实验报告助手需要展示实验要求、资料来源、证据卡片、数据表格、代码、日志、图表、Word 和 PPT 产物，Web 工作台最适合承载这些信息。
- 第一版优先跑通单用户完整闭环，不做 App、小程序、多人协作和自由拖拽工作流。
- 本地单用户形态能降低部署、账号、权限和云资源成本，更适合快速验证真实主路径。
- 第一版不做注册登录，不做在线多用户账号体系。
- 架构必须保留后续迁移到在线多用户版本的边界，但不得让在线化复杂度拖慢 V1。

## 2. 技术栈总览

| 层级 | V1 锁定选择 | 说明 |
| --- | --- | --- |
| 产品形态 | 本地单用户 Web MVP | 本地运行，浏览器访问工作台 |
| 前端 | TypeScript + React + Vite | 承载多工作区、表格、代码、日志、图表和下载入口 |
| 前端路由 | React Router | 管理工作台页面和项目内视图 |
| 接口状态 | TanStack Query | 管理请求、缓存、刷新和任务状态轮询 |
| 后端 | Python + FastAPI | 承载业务 API、任务编排、解析、LLM 网关和文件生成 |
| 数据校验 | Pydantic | 约束请求、响应和 LLM 结构化输出 |
| 数据访问 | SQLAlchemy + Alembic | 统一 ORM 和迁移，避免业务代码直接绑定数据库实现 |
| V1 数据库 | SQLite | 适用于本地单用户 MVP |
| 后续在线数据库 | PostgreSQL | 在线多用户版本再通过决策记录切换 |
| 文件存储 | 本地项目工作区目录 | 数据库保存文件元数据、哈希、版本和归属关系 |
| 后台任务 | 数据库任务表 + 独立 Worker 进程 | 长任务不阻塞普通 HTTP 请求 |
| AI 能力 | DeepSeek + 统一 LLM Gateway | 业务模块不得直接绑定模型名或供应商私有细节 |
| URL 采集 | HTTP 优先，Playwright 作为动态网页后备 | 不绕过登录、验证码、付费墙和访问控制 |
| 文档解析 | 按 PDF、Word、HTML、CSV、Excel 分适配器 | 第一版只支持明确范围内的稳定格式 |
| 数据分析 | 应用托管的固定 Python 分析环境 | 用户不需要手动配置 Python 环境或安装分析依赖 |
| Python 执行 | V1 受控独立进程，后续可升级容器沙箱 | 限定目录、时间、依赖、输出和访问范围 |
| Word 生成 | 模板驱动的 Word 渲染器 | 消费结构化交付物模型，不直接消费模型对话 |
| PPT 生成 | 母版驱动的 PPT 渲染器 | 与 Word 共用已确认大纲和追溯索引 |
| 架构形态 | 模块化单体 | 一个代码库、一套业务模型、一个本地数据源 |

具体包版本、目录模板和命令必须在代码实现阶段开始前基于官方文档复核，并写入实现计划或后续决策记录。

依赖版本、DeepSeek 模型和官方目录规范已在 [dependency-review.md](dependency-review.md) 完成初次复核。若超过 7 天仍未初始化依赖，或任一官方文档发生变化，必须重新复核。

## 3. 数据库选择规则

V1 锁定使用 SQLite，适用条件是：

- 本地单用户；
- 单机文件工作区；
- 中低并发；
- 主要目标是跑通端到端实验报告生成闭环。

不得把业务代码直接写死到 SQLite 私有能力上。数据库访问必须通过 ORM、迁移和仓储边界承载。

在线多用户版本触发以下任一条件时，必须重新评估并优先切换 PostgreSQL：

- 需要多用户账号和权限；
- 需要云端项目保存；
- 需要并发执行多个用户任务；
- 需要更强事务、连接管理、备份恢复或集中运维；
- SQLite 的锁、并发或文件部署限制成为阻碍。

## 4. 后台任务与 Worker

以下任务不得长期运行在普通 HTTP 请求中：

- URL 采集；
- PDF、Word、网页和表格解析；
- LLM 调用；
- 数据质量分析；
- Python 代码执行；
- 图表生成；
- Word 和 PPT 生成。

V1 采用：

```text
FastAPI API 进程
  -> 创建任务记录
  -> 数据库任务表
  -> 独立 Worker 进程领取并执行
  -> 更新任务状态、日志和产物索引
  -> 前端轮询任务状态
```

第一版不强制引入 Redis、Celery、RabbitMQ 或其他队列系统。任务量、并发和部署形态变化后，再通过任务适配器升级。

## 5. Python 受控执行环境

第一版采用：**应用托管的 Python 受控执行环境**。

用户体验目标：

- 普通用户只启动应用和打开 Web 工作台；
- 用户不需要单独配置 Python；
- 用户不需要手动安装 pandas、numpy、matplotlib 等分析依赖；
- 用户可以查看、修改并确认待执行代码。

V1 沙箱边界：

- 独立进程执行；
- 每次任务使用独立工作目录；
- 只允许访问被授权的数据版本目录和输出目录；
- 固定 Python 依赖白名单；
- 执行超时；
- 输出文件大小限制；
- 日志大小限制；
- 禁止 `shell=True`；
- 默认不暴露任意宿主机目录；
- 保存标准输出、错误输出、退出状态、执行耗时和产物清单。

V1 本地单用户版本的受控执行环境不是云端多用户强安全边界。若进入在线多用户版本，必须升级到容器级或等价隔离方案，并增加 CPU、内存、网络、子进程和密钥访问限制。

## 6. URL 采集策略

公开 URL 采集采用三级策略：

```text
第一级：直接 HTTP 下载
  -> 第二级：HTML 正文解析
  -> 第三级：Playwright 浏览器渲染
```

适用规则：

| 来源类型 | 处理方式 |
| --- | --- |
| PDF、CSV、Excel 直链 | HTTP 下载 |
| 普通公开文章网页 | HTTP 下载 + 正文解析 |
| JavaScript 动态网页 | Playwright 后备渲染 |
| 登录、验证码、付费内容 | 拒绝或要求用户自行上传 |
| 不支持的网站 | 明确提示失败原因，不尝试绕过限制 |

第一版不承诺支持任意网站，不做自动登录，不绕过访问控制。

## 7. Word 与 PPT 生成规则

大模型不得直接生成最终 `.docx` 或 `.pptx` 文件。

正确链路是：

```text
实验要求
+ 已确认任务单
+ 证据卡片
+ 数据版本
+ 执行记录
+ 图表产物
  -> 已确认实验大纲
  -> 结构化交付物模型
  -> Word 渲染器 / PPT 渲染器
  -> 可编辑 .docx / .pptx
```

Word 和 PPT 必须共用同一份已确认大纲、证据卡片、执行记录和图表索引。不得各自从模型临时上下文独立生成结论。

## 8. 工作流状态与过期传播

第一版必须有后端拥有的工作流状态机，前端只消费状态，不私自判断业务完成。

项目阶段至少应覆盖：

```text
DRAFT
REQUIREMENT_PARSED
REQUIREMENT_CONFIRMED
SOURCES_COLLECTED
EVIDENCE_CONFIRMED
DATASET_READY
ANALYSIS_PLANNED
ANALYSIS_CONFIRMED
EXECUTING
EXECUTION_FAILED
RESULT_CONFIRMED
OUTLINE_CONFIRMED
GENERATING
COMPLETED
```

长任务状态至少应覆盖：

```text
PENDING
RUNNING
SUCCEEDED
FAILED
CANCELLED
STALE
```

`STALE` 表示上游内容变化导致下游产物过期。例如用户修改实验要求后，旧分析方案、旧代码、旧图表、旧大纲和旧报告不得继续被视为有效结果。

## 9. 来源与产物追踪

第一版必须保存关键产物的来源链。

实验结论追踪链：

```text
报告结论
  -> 图表或表格
  -> ExecutionRun
  -> Python 代码版本
  -> 数据版本
  -> 原始数据集
```

资料性描述追踪链：

```text
理论描述
  -> EvidenceCard
  -> 原文位置
  -> 来源文件或 URL
```

这条追踪链是本项目区别于普通 AI 报告生成器的核心，不得在实现中省略。

## 10. 明确不采用的技术路线

第一版不采用：

- App 优先；
- 小程序优先；
- 纯后端 API 产品；
- Next.js 全栈作为主线后端；
- 微服务架构；
- Serverless 优先；
- 自由拖拽工作流引擎；
- 任意网站自动爬取；
- 自动登录受限平台；
- 一开始强依赖 Docker Desktop；
- 大模型直接生成最终 Word/PPT 文件。

这些选择不是永久否定，而是不进入 V1 主线。若后续需要引入，必须先更新立项文档、架构文档和决策记录。

## 11. 重新评估条件

出现以下情况时，必须重新评估技术栈：

- 项目从本地单用户转为在线多用户；
- 需要账号、权限、云端项目和多人协作；
- SQLite 出现并发、事务、备份或运维瓶颈；
- Python 执行需要处理不可信多用户代码；
- Word/PPT 模板复杂度超出固定版式；
- URL 采集范围扩大到需要登录、授权或复杂站点适配；
- V1 主路径无法通过验收标准；
- 官方框架主线或关键依赖发生重大变化。

重新评估不得只在对话中口头完成，必须写入 `dev-docs/decisions/`。
