# 决策 0005：锁定 V1 项目标识、登录边界与演示课题

> 日期：2026-06-16  
> 状态：已接受  
> 关联文档：[project-charter.md](../project-charter.md)、[architecture.md](../architecture.md)、[implementation-plan.md](../implementation-plan.md)

## 背景

技术栈已经在 `dev-docs/tech-stack.md` 与决策 0004 中锁定。进入代码实现阶段前，还需要收口项目目录命名、V1 登录边界，以及第一条端到端验收样例。

当前仍未批准进入代码实现阶段，因此本决策只更新项目真源，不创建业务代码、不安装依赖、不初始化技术框架。

## 决策

V1 锁定以下边界：

- 项目规范目录名：`lab-report-assistant`。
- 当前 Codex 会话仍运行在旧工作区路径 `C:\Users\爹\Documents\VibeCoding`，物理目录迁移需在不打断当前会话的安全时机执行。
- 第一版不做注册登录。
- 第一版不做在线多用户账号体系。
- 第一版首个标准演示课题为：**胃病数据分析**。
- 首个标准演示课题的数据来源仍需后续确认。

## 原因

- `lab-report-assistant` 符合英文加中划线的项目命名规范，也能直接表达产品边界。
- 本地单用户 V1 不需要注册登录；加入账号体系会提前引入权限、会话、用户数据隔离和云端同步复杂度。
- “胃病数据分析”已在立项文档中作为典型场景出现，适合作为第一条端到端验收样例。
- 样例数据来源会影响数据字段、分析方案、图表和报告结构，必须单独确认，不能由智能体自行假设。

## 影响

- `project-charter.md` 中的 V1 登录边界、部署形态和首个演示课题不再属于待确认事项。
- `implementation-plan.md` 的任务 0 只剩样例数据来源、开发命令与代码阶段批准等执行前事项。
- 后续实现不得加入注册登录页面、用户注册流程、多人账号管理或云端项目协作。
- V1 验收用例应围绕“胃病数据分析”展开。

## 重新评估条件

出现以下情况时，必须新增或更新决策记录：

- 项目从本地单用户改为在线多用户；
- 需要账号、登录、权限、云端保存或多人协作；
- 首个标准演示课题更换；
- 样例数据来源无法合法获取或无法支撑 V1 验收闭环；
- 物理项目目录迁移会影响 git、IDE、Codex 会话或文档链接。
