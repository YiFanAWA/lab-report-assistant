# 决策 0011：每版完成后进行 git 提交与上传收口

## 状态

已记录。

## 日期

2026-06-17

## 背景

项目负责人要求：“每完成一版使用 git 上传一次做版本控制”。此前项目宪法已有基础 git 卫生规则，例如禁止 `git add .`、只提交相关文件、忽略本地运行文件，但没有明确规定每个版本或开发切片完成后的提交与上传节奏。

## 决策

- 每完成一版或一个已确认开发切片，必须进行一次 git 版本控制收口。
- 收口必须发生在实现完成、验收通过、文档回写完成、项目负责人确认收口之后。
- 收口流程为：git 状态复核 -> diff 复核 -> 精确 stage -> 中文 commit -> push 到已配置远程仓库。
- 如果远程仓库未配置、鉴权失败或网络不可用，必须保留本地 commit，并在验收记录或交接中说明未上传原因。
- 默认不自动创建 tag；只有项目负责人明确要求发布版本号时，才创建并上传 tag。

## 影响范围

- 根目录 `AGENTS.md` 新增“版本收口上传规则”。
- `dev-docs/commands.md` 新增版本收口命令索引。
- `dev-docs/README.md` 新增本决策记录入口。
- 后续 SPEC 0002 收口、SPEC 0003 及以后所有版本完成时，都必须执行该规则。

## 压力测试

场景一：后续 agent 完成 SPEC 0002 并准备进入下一切片。正确行为是先要求项目负责人确认 SPEC 0002 收口，然后按验收证据、文档回写、精确提交、远程上传的顺序完成版本控制；不得直接进入下一切片。

场景二：工作区存在 `.obsidian/`、`.claude/`、`.codex/`、`memory/`、`.venv/`、`dist/`、`.tmp/` 或本地数据库文件。正确行为是确认这些文件留在 git 外，只 stage 本版相关源码和文档；禁止使用 `git add .`。

场景三：当前没有远程仓库或 push 失败。正确行为是保留本地 commit，记录未上传原因，并向项目负责人说明需要配置远程仓库或处理鉴权；不得谎称已上传。
