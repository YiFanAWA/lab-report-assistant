# 决策 0004：锁定 V1 技术栈

> 日期：2026-06-16  
> 状态：已接受  
> 关联文档：[project-charter.md](../project-charter.md)、[architecture.md](../architecture.md)、[tech-stack.md](../tech-stack.md)

## 背景

项目已完成立项确认，并进入架构与开发计划阶段。项目负责人确认第一版优先采用本地单用户版本，并认可“应用托管的 Python 受控执行环境”作为 V1 的执行体验和安全边界。

当前仍未批准进入代码实现阶段，因此本决策只锁定技术方向，不创建业务代码、不安装依赖、不初始化技术框架。

## 决策

V1 技术栈锁定为：

- 产品形态：本地单用户 Web MVP 工作台。
- 前端：TypeScript + React + Vite。
- 前端路由：React Router。
- 接口状态：TanStack Query。
- 后端：Python + FastAPI。
- 数据校验：Pydantic。
- 数据访问：SQLAlchemy + Alembic。
- 数据库：V1 使用 SQLite；在线多用户版本再评估并优先切换 PostgreSQL。
- 文件存储：本地项目工作区目录，数据库保存文件元数据、哈希、版本和归属关系。
- 后台任务：数据库任务表 + 独立 Worker 进程。
- AI 能力：统一 LLM Gateway，不在业务模块直接绑定模型供应商。
- URL 采集：HTTP 优先，HTML 正文解析其次，Playwright 作为动态网页后备。
- Python 执行：应用托管的受控执行环境，V1 采用独立进程、独立工作目录、固定依赖白名单、执行超时、文件访问范围和输出大小限制。
- Word/PPT：通过结构化交付物模型和模板渲染器生成，不直接由大模型生成最终文件。
- 架构形态：模块化单体。

## 原因

- Web MVP 最适合承载多步骤工作台、文件、表格、代码、日志、图表和下载交付物。
- 本地单用户版本能优先验证端到端闭环，避免账号、权限、云部署和多租户复杂度提前进入。
- Python + FastAPI 与文档解析、数据分析、LLM 调用、Python 执行和 Word/PPT 生成生态匹配。
- SQLite 适合本地单用户 MVP，但必须通过 ORM 和迁移工具保留后续切换 PostgreSQL 的边界。
- 长任务需要 Worker 承担，避免 HTTP 请求长期阻塞。
- Python 执行不能只靠独立目录，必须形成可描述、可验证、可升级的受控执行环境。
- 结构化交付物模型能保证 Word 和 PPT 来自同一份已确认大纲、证据和执行结果。

## 不采用

V1 不采用以下路线：

- App 或小程序优先；
- 微服务；
- Serverless 优先；
- 自由拖拽工作流引擎；
- 任意网站自动爬取；
- 自动登录受限平台；
- 一开始强依赖 Docker Desktop；
- 大模型直接生成最终 Word/PPT 文件。

## 影响

- `dev-docs/tech-stack.md` 成为技术栈真源。
- `dev-docs/architecture.md` 需要以本决策为准同步 Worker、Python 受控执行环境、状态机和追踪链约束。
- `dev-docs/implementation-plan.md` 的任务 0 不再需要继续确认技术主线，但仍需要代码阶段批准。
- 后续代码实现必须映射到本决策，不得私自切换框架、数据库、执行环境或产品形态。

## 重新评估条件

出现以下情况时，必须新增或更新决策记录：

- 项目转向在线多用户版本；
- 需要账号、权限、云端项目或多人协作；
- SQLite 无法满足并发、事务、备份或运维要求；
- Python 执行需要运行不可信多用户代码；
- Word/PPT 模板复杂度超出固定版式；
- URL 采集范围超出公开可访问资源；
- 官方框架主线或关键依赖发生重大变化。
