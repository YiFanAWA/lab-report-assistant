## 0. 立项约束与智能体行为准则

本项目在立项阶段遵守以下约束：

1. **阶段门禁优先**  
   立项阶段只完成产品边界、工作流、业务对象、初步技术方向、里程碑和验收标准。代码执行必须在项目负责人明确批准后开始；当前代码阶段已由决策 0007 承接为正式执行，SPEC 0002 已确认收口，下一切片必须先确认 SPEC。

2. **多追问、少假设**  
   已由项目负责人明确的内容标记为“已确认”；尚未明确但不阻塞立项的内容标记为“待验证”或“建议默认值”，不得将推测写成既定事实。

3. **产品价值优先**  
   所有技术设计必须服务于“帮助高校本科生理解实验要求、完成数据实验并生成可审阅交付物”这一目标，不为展示技术而增加功能。

4. **固定工作流优先**  
   第一版采用固定、可检查、可回退的工作流，不实现自由拖拽式工作流编排。

5. **人工确认优先**  
   大模型负责理解、拆解、建议和生成，不替用户做不可逆决定。实验要求、分析方案、代码执行结果和最终交付物均保留人工确认节点。

6. **证据与结果可追溯**  
   资料性结论应关联公开来源；实验性结论必须来自真实执行结果。未执行的实验不得描述为已完成。

7. **合法合规采集**  
   不绕过登录、验证码、付费墙和访问控制，不自动登录知网等受限制平台，不承诺支持任意网站。

8. **文档是真源**  
   后续开发、任务拆解和技术选型不得偏离本立项文档。发生方向变化时，必须同步更新版本、决策记录和验收标准。

# 1. 项目定义

## 1.1 项目名称

**实验报告助手**

## 1.2 一句话描述

面向高校本科生的实验材料采集、实验要求拆解、数据分析执行与交付物生成工作流。

## 1.3 面向用户

第一版核心用户为：

- 高校本科生；
- 已获得老师指定课题或实验任务；
- 需要查找公开资料、处理实验数据、完成数据分析与可视化；
- 最终需要提交 Word 实验报告和/或 PPT 汇报。

## 1.4 典型使用场景

老师给出一个明确课题，例如“胃病数据分析”或其他独立数据分析主题，并提供实验要求文件。学生需要：

1. 理解老师真正要求完成的任务；
2. 查找公开论文、网页和数据集；
3. 获取并保存可合法访问的资料；
4. 阅读论文核心概念和实验方法；
5. 上传或获取 CSV、Excel 等数据；
6. 完成数据预览、清洗、统计分析和可视化；
7. 参考论文方法进行简易复刻；
8. 根据实验要求生成 Word 报告和 PPT；
9. 检查资料来源、实验结果和交付内容后再提交。

## 1.5 解决的核心问题

本项目重点解决以下问题：

- 实验要求通常是自然语言，学生难以转化为明确任务；
- 搜集到的论文、网页、数据和实验要求彼此割裂；
- 学生难以判断论文应当“参考到什么程度”；
- 数据清洗、统计分析、图表与实验报告之间缺少统一工作流；
- 普通大模型可能生成无法追溯、未经执行或不符合要求的内容；
- Word 报告和 PPT 内容容易不一致；
- 生成过程缺少检查点，错误会一直传递到最终交付物。

## 1.6 为什么值得做

“实验报告助手”不是单纯的网页爬虫，也不是普通的 AI 代写工具。它试图把以下环节连接为可审阅工作流：

```text
明确课题
→ 理解实验要求
→ 获取公开资料
→ 提取论文证据
→ 获取并理解数据
→ 生成实验方案
→ 执行 Python 分析
→ 审查真实结果
→ 生成 Word 与 PPT
```

其核心价值是：**让实验要求、资料证据、数据处理、代码执行、图表结果和最终交付物保持一致并可追溯。**

# 2. 第一版核心目标

## 2.1 V1 核心目标

第一版需要完成一条可实际运行的主路径：

> 用户创建一个实验项目，输入明确课题，上传老师的实验要求 Word 文件或粘贴要求，系统使用大模型将要求拆解为结构化任务；用户添加公开 URL 或上传本地资料和数据；系统完成资料解析、数据预览、分析方案生成和受控 Python 执行；用户确认结果后，系统分别生成可编辑的 Word 实验报告和 PPT 汇报文件。

## 2.2 第一版完成后的可见结果

用户至少能够获得：

- 一份结构化实验需求任务单；
- 一组可追溯的论文或网页资料；
- 一份数据字段与质量概览；
- 一份可确认的数据分析方案；
- 一份实际运行过的 Python 分析结果；
- 若干由真实数据生成的图表；
- 一份可编辑的 `.docx` 实验报告；
- 一份与报告内容一致的 `.pptx` 汇报文件；
- 一份来源和执行记录。

## 2.3 第一版成功标准

第一版不以“支持所有实验”为成功标准，而以以下标准为准：

- 至少有一类典型数据分析实验能够完整跑通；
- 工作流中每个阶段都有明确输入、输出和状态；
- 生成内容能够区分“资料事实”“模型建议”“真实实验结果”；
- 用户能够在关键阶段确认、修改、重新执行或回退；
- 最终 Word 和 PPT 与已确认的大纲及实验结果保持一致。

# 3. 功能范围与边界

## 3.1 最小可行产品必须实现

### 3.1.1 实验项目空间

每次实验建立独立项目，至少保存：

- 项目名称；
- 明确课题；
- 实验要求；
- 来源资料；
- 数据文件；
- 需求拆解结果；
- 实验大纲；
- 分析方案；
- Python 代码；
- 执行日志；
- 数据表格；
- 图表；
- Word 报告；
- PPT 文件；
- 版本和状态。

### 3.1.2 实验要求输入

第一版支持：

- 上传老师提供的 Word 实验要求文件；
- 粘贴实验要求文本；
- 用户补充说明。

三类输入合并为一份实验要求来源，但必须保留原始内容。

### 3.1.3 大模型需求拆解

大模型将自然语言要求转化为结构化任务单，至少提取：

- 课题；
- 实验类型；
- 研究对象；
- 必须完成的任务；
- 推荐任务；
- 可选任务；
- 明确禁止或超出范围的任务；
- 数据要求；
- 分析方法要求；
- 图表要求；
- Word 结构要求；
- PPT 要求；
- 参考文献要求；
- 代码要求；
- 验收条件；
- 未明确项；
- 需要用户确认的问题。

对于缺失信息，系统提供建议默认值，但不得伪装成老师的原始要求。

### 3.1.4 简易论文复刻分级

系统根据实验要求判断复刻层级，并允许用户在支持范围内调整：

- **L0：不复刻**  
  论文只用于背景、概念和参考文献。

- **L1：方法参考**  
  参考论文的数据处理或分析思路，使用其他公开数据或用户数据重新完成实验。

- **L2：局部复现**  
  在数据、方法和条件允许时，复现论文中的一个或少量处理步骤、统计结果或图表。

- **L3：完整复现**  
  运行论文原始代码、配置复杂依赖并复现全部指标。第一版不支持。

系统不得把 L1 或 L2 描述为完整论文复现。

### 3.1.5 公开 URL 资料获取

第一版以公开 URL 为核心入口，支持以下优先类型：

- 普通公开网页；
- 可直接访问的论文详情页；
- 可直接下载的公开 PDF；
- 公开数据集页面；
- 可直接访问的 CSV、Excel 文件。

每条来源至少记录：

- 原始 URL；
- 页面或文件标题；
- 来源类型；
- 采集时间；
- 采集状态；
- 原始内容或本地文件；
- 内容摘要；
- 可定位的来源信息。

### 3.1.6 本地文件上传

第一版辅助支持：

- PDF；
- Word；
- CSV；
- Excel；
- TXT。

图片识别、扫描件 OCR 和更多格式不作为第一版必须项，除非后续决策记录明确加入。

### 3.1.7 资料解析与证据卡片

系统从论文和网页中提取：

- 研究背景；
- 核心问题；
- 研究对象；
- 数据来源；
- 实验方法；
- 数据处理步骤；
- 主要结果；
- 结论；
- 局限性；
- 可参考或可复刻部分。

关键内容以证据卡片保存，至少包含：

- 内容摘要；
- 来源文件或 URL；
- 页码、章节或网页位置；
- 证据类型；
- 是否已经用户确认。

### 3.1.8 数据工作区

支持以下主流程：

```text
上传或下载 CSV/Excel
→ 数据预览
→ 字段识别
→ 数据类型判断
→ 缺失值检查
→ 重复值检查
→ 异常值提示
→ 清洗方案
→ 分析方案
→ 图表方案
→ Python 执行
→ 结果审阅
```

### 3.1.9 受控 Python 执行环境

第一版使用固定、受限制、可重复运行的数据分析环境。

主要支持：

- CSV、Excel 读取；
- 常见数据清洗；
- 描述性统计；
- 分组统计；
- 相关性分析；
- 基础统计检验；
- 基础回归或分类；
- 常见数据可视化；
- 表格和图片导出。

大模型可以辅助生成和修改代码，但代码在执行前应可见，执行后必须保存：

- 实际代码；
- 输入数据版本；
- 执行状态；
- 标准输出；
- 错误日志；
- 生成表格；
- 生成图表；
- 执行时间。

### 3.1.10 实验大纲

系统根据以下内容生成统一实验大纲：

```text
实验要求
+ 已确认任务单
+ 资料证据
+ 数据字段
+ 分析方案
+ 真实执行结果
```

大纲必须由用户确认后，才进入最终交付物生成阶段。

### 3.1.11 Word 与 PPT 生成

- Word 根据老师提供的要求或模板结构生成；
- PPT 从同一份已确认大纲提炼；
- Word 注重完整过程、数据、代码、图表和结论；
- PPT 注重问题、方法、关键图表、主要发现和总结；
- 两者不得使用相互矛盾的数据或结论；
- 所有文件生成后均允许用户审阅并重新生成。

## 3.2 第一版明确不做

- 不绕过登录、验证码、付费限制和访问控制；
- 不自动登录知网等受限制平台；
- 不承诺支持所有网站；
- 不做注册登录；
- 不做在线多用户账号体系；
- 不做手机 App；
- 不做小程序；
- 不支持复杂科研论文完整复现；
- 不自动配置任意论文的复杂运行环境；
- 不做多人协作；
- 不做自由拖拽工作流；
- 不做论文投稿级排版；
- 不保证内容可以未经检查直接提交；
- 不生成虚构实验结果；
- 不把失败实验改写为成功；
- 不提供医疗诊断或治疗建议；
- 不将教学数据分析结论包装为临床结论。

## 3.3 已讨论但放弃或推迟的方向

- “输入一个网站后直接一键生成完整报告”被放弃，改为可审阅工作流；
- “第一版支持所有学科实验”被推迟，先聚焦数据分析与可视化类实验；
- “直接完整复现任意论文”被排除；
- “第一版自由编排节点”被排除；
- “自动搜索并获取所有论文全文”被排除；
- “系统自动决定所有数据清洗动作”被排除。

# 4. 用户路径与页面流程

## 4.1 主用户路径

```text
1. 创建实验项目
2. 输入老师指定课题
3. 上传或粘贴实验要求
4. 大模型拆解要求
5. 用户确认任务单与复刻层级
6. 添加公开 URL
7. 上传辅助资料
8. 系统解析资料并生成证据卡片
9. 获取或上传 CSV/Excel
10. 查看字段与数据质量
11. 生成实验大纲和分析方案
12. 用户确认分析方案
13. 生成或修改 Python 代码
14. 在受控环境中执行
15. 查看日志、表格和图表
16. 用户确认可用结果
17. 更新并确认实验大纲
18. 生成 Word
19. 生成 PPT
20. 审阅、修改、重新生成并下载
```

## 4.2 建议页面结构

第一版采用：

> **多区域工作台作为整体布局，步骤向导作为操作顺序。**

建议页面：

1. 项目列表页；
2. 创建项目页；
3. 实验要求工作区；
4. 资料来源工作区；
5. 证据卡片工作区；
6. 数据工作区；
7. 实验方案工作区；
8. Python 执行工作区；
9. 结果与图表工作区；
10. 大纲工作区；
11. Word 预览与生成页；
12. PPT 预览与生成页；
13. 项目记录与版本页。

建议工作台布局：

```text
左侧：项目步骤与状态
中间：当前工作区
右侧：AI 助手、来源信息、待确认问题
```

## 4.3 关键人工确认点

以下阶段不得默认无感跳过：

- 实验需求任务单确认；
- 论文复刻层级确认；
- 数据清洗与分析方案确认；
- Python 执行结果确认；
- 实验大纲确认；
- Word 和 PPT 最终审阅。

## 4.4 需求变更处理

用户修改实验要求后，系统必须标记受影响内容，例如：

- 任务单需要重新生成；
- 资料检索策略可能失效；
- 数据分析方案需要重新确认；
- 已执行代码可能需要重跑；
- 已生成图表可能失效；
- Word 和 PPT 需要重新生成。

旧版本不能直接覆盖，应保留版本或至少保留变更记录。

# 5. 核心业务对象与数据流

## 5.1 核心业务对象

### 用户

第一版核心使用者。账号模式尚未锁定，但业务模型中保留用户概念。

### 实验项目

一次完整实验的工作空间，是其他业务对象的归属主体。

### 课题

老师指定的明确研究主题，例如胃病、肺病或其他数据分析课题。

### 实验要求来源

老师的 Word 文件、粘贴文本和用户补充说明。

### 结构化任务单

大模型解析后的任务集合，包含必须项、推荐项、可选项、未知项和验收条件。

### 来源资料

公开网页、论文页面、PDF、数据集页面或本地上传文件。

### 证据卡片

从资料中提取的可引用事实、方法、结论和来源位置。

### 数据集

公开获取或用户上传的 CSV、Excel 等数据。

### 数据版本

原始数据、清洗后数据和各次修改版本。

### 分析方案

清洗步骤、统计方法、可视化方案和预期输出。

### 代码任务

为某个分析步骤生成或编辑的 Python 代码。

### 执行记录

一次真实 Python 执行，包括代码、数据版本、日志、表格和图表。

### 实验大纲

连接要求、证据、分析和交付物的统一结构。

### 交付物

Word、PPT 及其生成版本。

### 变更记录

记录要求、方案、大纲和交付物的重要变化。

## 5.2 核心关系

```text
User
  └─ Project
      ├─ Topic
      ├─ RequirementSource
      │   └─ RequirementPlan
      ├─ Source
      │   └─ EvidenceCard
      ├─ Dataset
      │   └─ DatasetVersion
      ├─ AnalysisPlan
      │   └─ CodeTask
      │       └─ ExecutionRun
      ├─ Outline
      ├─ Deliverable
      └─ ChangeRecord
```

## 5.3 数据流

```text
课题 + 原始实验要求
→ 大模型需求拆解
→ 结构化任务单
→ 公开 URL / 本地文件
→ 原始资料本地保存
→ 解析与证据卡片
→ 数据集与字段信息
→ 分析方案
→ Python 代码
→ 受控执行
→ 真实表格和图表
→ 已确认实验大纲
→ Word / PPT
→ 人工审阅
```

## 5.4 数据真实性规则

- 资料事实必须关联来源；
- 实验结果必须关联执行记录；
- 图表必须关联生成它的数据版本和代码；
- 模型建议必须标记为建议；
- 未确认信息必须标记为待确认；
- 未执行步骤不得写成已完成；
- 失败记录不得被静默删除。

# 6. 技术路线初步判断

> 本节只确定架构方向，不锁定具体框架。具体技术栈需要在立项确认后单独评审并写入技术选型文档。

## 6.1 平台形态

第一版建议采用 **Web MVP**。

原因：

- 适合多步骤工作台；
- 便于展示数据表、代码、日志和图表；
- 便于生成和下载 Word、PPT；
- 便于在服务端统一执行受控 Python；
- 后续可扩展为在线服务或本地部署。

## 6.2 初步模块划分

### 前端工作台

负责：

- 项目导航；
- 文件与 URL 输入；
- 任务单确认；
- 数据表预览；
- 分析方案审阅；
- 代码查看和编辑；
- 运行状态展示；
- 图表和交付物预览。

### 后端业务服务

负责：

- 项目管理；
- 文件管理；
- 工作流状态；
- 任务编排；
- 版本记录；
- Word 和 PPT 生成；
- 权限与访问控制。

### 数据源适配层

负责：

- 公开 HTML；
- 公开 PDF；
- 公开 CSV/Excel；
- 本地上传文件。

采用可扩展适配器设计，但第一版只实现少量稳定来源类型。

### 文档解析层

负责：

- Word 要求解析；
- PDF 文本解析；
- 网页正文提取；
- 表格和元数据提取；
- 来源位置保留。

### 大模型网关

负责统一调用大模型，并区分任务：

- 要求拆解；
- 资料摘要；
- 证据提取；
- 复刻层级判断；
- 检索词建议；
- 分析方案生成；
- Python 代码辅助；
- 结果解释；
- 大纲生成；
- Word/PPT 内容生成；
- 完整性检查。

大模型输出应优先采用结构化格式，并经过规则校验。

### Python 执行沙箱

负责：

- 固定依赖环境；
- 资源和时间限制；
- 文件访问限制；
- 网络访问策略；
- 执行日志；
- 结果文件收集；
- 失败隔离；
- 可重复运行。

### 数据存储

至少区分：

- 项目元数据；
- 原始文件；
- 解析文本；
- 数据集版本；
- 代码；
- 日志；
- 图表；
- Word/PPT；
- 来源与变更记录。

## 6.3 技术原则

- 先实现单体 Web MVP，不做过早微服务化；
- 爬取能力采用适配器扩展，不绑定单个平台；

- 大模型调用经过统一网关；
- 所有关键产出保存版本和来源；

# 7. 差异化与优势点

## 7.1 从“生成器”转为“证据化工作流”

普通工具通常直接生成报告。本项目要求每一步都有输入、输出、状态和来源。

## 7.2 实验要求先拆解再执行

系统先理解老师真正要求什么，再决定：

- 查什么资料；
- 找什么数据；
- 是否需要论文复刻；
- 分析到什么程度；
- 生成哪些图表；
- 最终交付物如何组织。

## 7.3 简易论文复刻分级

通过 L0—L3 分级，避免“参考论文方法”和“完整论文复现”混为一谈。

## 7.4 真实 Python 执行

报告中的实验结果来自受控环境真实执行，而不是仅由大模型编写描述。

## 7.5 资料证据与执行结果双追溯

- 理论和背景可追溯到来源资料；
- 实验结论可追溯到数据、代码和执行记录。

## 7.6 Word 与 PPT 共用统一大纲

减少报告与汇报内容不一致的问题。

## 7.7 可检查、可回退、可重新运行

用户不是被动接受最终答案，而是在关键节点控制工作流。

# 10. 风险、限制与验证计划

## 10.1 范围风险

项目同时涉及网页采集、文档解析、大模型、数据分析、代码执行、Word 和 PPT，范围较大。

应对原则：

- 第一版只支持数据分析与可视化类实验；
- 先跑通一个典型课题；
- 每个版本只增加一个主要闭环；
- 不用“支持所有学科”作为第一版目标。

## 10.2 网页与版权风险

公开访问不等于可以任意复制、传播或二次发布。

应对原则：

- 记录来源；
- 遵守网站访问规则；
- 不绕过限制；
- 优先保存必要内容和引用信息；
- 不将受版权保护的全文作为产品公开内容分发。

## 10.3 大模型幻觉风险

大模型可能错误理解要求、编造来源或生成不适用代码。

应对原则：

- 结构化输出；
- 来源验证；
- 人工确认；
- 代码真实执行；
- 结果规则检查；
- 明确标记建议与事实。

## 10.4 代码执行安全风险

大模型生成代码可能访问不应访问的资源、消耗大量资源或破坏文件。

应对原则：

- 隔离执行；
- 固定依赖；
- 时间和资源限制；
- 文件目录限制；
- 默认限制网络；
- 保存完整日志；
- 禁止危险系统操作。

## 10.5 学术诚信风险

产品可能被用于未经理解直接提交作业。

应对原则：

- 定位为实验理解和辅助工作流；
- 保留代码、步骤和来源；
- 强制人工审阅；
- 不承诺直接提交；
- 不虚构数据和结果；
- 可以提供解释和学习提示。

## 10.6 医学主题风险

胃病、肺病等医学课题可能被误解为诊断工具。

应对原则：

- 只处理教学和公开数据分析；
- 不生成个体诊断；
- 不提供治疗建议；
- 不把相关性描述为因果或临床结论；
- 明确数据和方法限制。

## 10.7 待验证的产品假设

以下内容尚未通过真实用户验证：

- 学生最痛苦的环节是否是要求拆解；
- 学生是否愿意逐步确认，而不是追求一键生成；
- L0—L2 分级是否容易理解；
- 公开 URL 是否比文件上传更常用；
- Word 和 PPT 是否都属于高频需求；
- 代码可编辑是否是第一版必要功能；
- 用户是否需要账号和云端保存；
- 老师提供的 Word 模板复杂度。

这些假设需要在原型或首批用户测试中验证。
