## 第二部分：V1.1.0 版本功能迭代规划

### 一、版本定位

**V1.1.0 = V1.0 + 真实 LLM 接入 + 部署完善 + 测试补全 + 小幅增强**

V1.1.0 不是大版本跳跃，而是 V1.0 的增量改进：
- 不改变产品边界（仍是本地单用户 Web MVP）
- 不改变架构主线（仍是唯一 owner + API 适配 + 前端接线）
- 不引入新的业务模块
- 聚焦于填补 V1.0 的功能缺口和提升候选质量

### 二、V1.1.0 功能迭代清单

#### 核心目标 1：真实 DeepSeek LLM 接入（SPEC 0007）

**优先级：** 🔴 最高  
**预计工作量：** 3-4 天  
**解决限制：** L1、A1

**范围：**
- 接入真实 DeepSeek API（通过 LLM Gateway 统一接入，不写死模型名）
- 替换以下 LocalRule 提供者为 LLM 提供者：
  - `RequirementDraftProvider`（实验要求拆解）
  - `EvidenceCardProvider`（证据卡片提取）
  - `AnalysisPlanProvider`（分析方案候选）
  - `CodeTaskProvider`（Python 代码生成）
  - `OutlineProvider`（实验大纲生成）
- 保留 LocalRule 作为后备（LLM 不可用时降级）
- 新增环境变量：`DEEPSEEK_API_KEY`（真实密钥）
- 新增 LLM 调用日志和成本记录
- Pydantic 结构化输出校验（防止 LLM 幻觉）

**不做：**
- 不接入其他 LLM 供应商（V1.1.0 只接 DeepSeek）
- 不做流式输出（V1.1.0 保持异步任务模式）
- 不做 LLM 调用缓存优化

**验收：**
- 真实 DeepSeek 调用成功，5 个提供者全部替换
- LLM 不可用时自动降级到 LocalRule
- 结构化输出校验通过
- 新增 LLM 调用测试（mock + 真实调用可选）

#### 核心目标 2：部署文档和运维指南（V1.0 遗留）

**优先级：** 🟡 高  
**预计工作量：** 1-2 天  
**解决限制：** L7

**范围：**
- 根目录 `README.md`（面向首次使用者）
- 包含内容：
  - 环境要求（Python 3.10+、Node 18+、SQLite）
  - 后端启动步骤（venv 创建、pip install、alembic upgrade、uvicorn 启动）
  - 前端启动步骤（npm install、npm run dev）
  - Worker 启动步骤（python -m worker）
  - 环境变量清单（DATABASE_URL、PROJECT_DATA_ROOT、DEEPSEEK_API_KEY 等）
  - 常见排障（端口冲突、SQLite 锁定、Worker 不领取任务）
- 从零启动验证（按文档执行一次，确认可复现）

**不做：**
- V1.1.0 暂不 Docker 化（推迟到 V1.2.0）
- 不编写 CI/CD 配置

#### 核心目标 3：前端测试覆盖补全

**优先级：** 🟡 高  
**预计工作量：** 2-3 天  
**解决限制：** L6

**范围：**
- 为以下 7 个工作区视图补充 RTL 组件测试：
  - `RequirementWorkspaceView`（实验要求）
  - `SourceWorkspaceView`（公开资料）
  - `EvidenceWorkspaceView`（证据卡片）
  - `DatasetWorkspaceView`（数据集）
  - `AnalysisWorkspaceView`（分析方案）
  - `OutlineWorkspaceView`（大纲）
  - `DeliverableWorkspaceView`（交付物）
- 为以下 API 模块补充 api.test.ts：
  - `features/requirements/api.ts`
  - `features/sources/api.ts`
  - `features/evidence/api.ts`
  - `features/datasets/api.ts`
  - `features/analysis/api.ts`
  - `features/outlines/api.ts`
- 目标：前端测试从 37 个增加到 ~200 个

**不做：**
- 不追求 100% 覆盖率
- 不引入 E2E 测试框架（Playwright/Cypress）
- 不引入快照测试

#### 增强目标 4：Word 模板支持 ✅ 已完成

**优先级：** 🟢 中  
**预计工作量：** 1-2 天  
**解决限制：** L2、P1  
**状态：** 已完成实现与测试验收（SPEC 0010，后端 623 passed + 前端 411 passed）

**实际实现范围：**
- 支持项目级上传 `.docx` 模板文件（每个项目可绑定不同模板，覆盖式存储）
- 读取模板中的 Jinja2 风格占位符（`{{project_name}}`、`{{project_topic}}`、`{{generated_date}}` 等封面变量 + `{{section_title}}`、`{{section_content}}` 等章节循环变量）
- 章节循环渲染 `{{#sections}}...{{/sections}}`（文本重建方式）
- 无模板时保持现有默认模板行为
- 新增 `WORD_TEMPLATE_MAX_SIZE_BYTES` 环境变量（默认 5MB）
- 模板渲染失败时降级到默认渲染 + warning 日志

**不做：**
- 不支持复杂样式继承
- 不支持 VBA 宏
- 不支持老师特定模板的自动识别
- 不支持模板预览（推迟到 V2.0）

#### 增强目标 5：PPT 页数和样式配置

**优先级：** 🟢 中  
**预计工作量：** 1 天  
**解决限制：** L3、P2  
**状态：** SPEC 0011 已编写，待项目负责人确认后进入实现

**范围：**
- PPT 生成时支持配置：目标页数（5-20 页）、主题色（6 个预设色板）、是否包含图表（全局开关）
- 前端大纲工作区新增 PPT 配置表单
- 后端 PptRenderer 接收 config 参数
- 配置不持久化，每次生成时传参

**不做：**
- 不做 PPT 动画
- 不做复杂母版编辑
- 不做在线 PPT 预览
- 不做任意 hex 色值输入（使用预设色板）
- 不做每张图表单独配置（使用全局开关）

#### 增强目标 6：数据集保留周期配置

**优先级：** 🟢 低  
**预计工作量：** 0.5 天  
**解决限制：** P6

**范围：**
- 新增 `DATA_RETENTION_DAYS` 环境变量（默认 0 = 永久）
- 提供清理脚本（手动执行，不自动清理）
- 文档说明清理策略

**不做：**
- 不做自动清理定时任务
- 不做项目级保留策略

### 六、V1.1.0 验收标准

1. 真实 DeepSeek 调用成功，5 个提供者全部替换，LLM 不可用时降级到 LocalRule
2. 部署文档完整，从零启动可复现

4. Word 模板支持（有模板用模板，无模板用默认）

### 七、风险评估

| 风险 | 影响 | 缓解措施 |
| --- | --- | --- |
| DeepSeek API 不稳定 | LLM 调用失败 | 保留 LocalRule 降级 + 重试机制 |
| LLM 幻觉导致结构化输出校验失败 | 候选质量下降 | Pydantic 严格校验 + 人工确认点 |
| DeepSeek API 成本 | 用户成本增加 | 记录调用次数和 token 消耗 |
| 前端测试补全工作量大 | 开发周期延长 | 按模块优先级补全，非核心模块可推迟 |
| Word 模板兼容性 | 模板解析失败 | 限制支持的占位符类型，提供错误提示 |
